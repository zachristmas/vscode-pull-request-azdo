**artifactId formats (do not confuse - ROADMAP Section 4):**

- Policy evaluations (POL-01): `vstfs:///CodeReview/CodeReviewId/{projectId}/{pullRequestId}` (Microsoft Learn, PolicyEvaluationRecord.ArtifactId remarks). This is **not** the PR's own `artifactId` field.

## 1. Local TypeScript interfaces for untyped policy payloads

`PolicyConfiguration.settings` and `PolicyEvaluationRecord.context` are `any` (`PolicyInterfaces.d.ts:37,79`). Two layers:

- **Raw wire shapes** (extension host only) - new file `src/azdo/policyTypes.ts`. Never imported by webview code.
- **Webview DTO** - added to `src/azdo/interface.ts` (the file `webviews/common/cache.ts` already imports from), converted host-side in `src/azdo/utils.ts` following the `convertIdentityRefWithVoteToReviewer` precedent (`utils.ts:56`).

### 1.1 `src/azdo/policyTypes.ts` - raw settings/context shapes


**Provenance / assumption flags:**

- Field names for settings come from Microsoft Learn branch-policy docs and the `az repos policy` CLI flags (same wire names). They are **docs-derived, not typings-derived** - the d.ts offers nothing (`settings: any`). Marked assumption: verify each shape once against a live evaluation payload from the test org during POL-01 implementation (dump `JSON.stringify(record.configuration.settings)` behind `Logger.debug` and compare).
- `BuildPolicyEvaluationContext` matches ROADMAP Section 2.2's field list (buildId/buildDefinitionId/isExpired/buildIsNotCurrent). Same live-payload verification applies. `buildId` is absent until a build has been queued (e.g. `manualQueueOnly` policies before first queue).
- `useSquashMerge` on the merge-strategy policy is the legacy single-flag settings form; orgs with old policies still return it. Treat `useSquashMerge: true` as `allowSquash: true` and all others false.

### 1.2 Kind detection: shape-first, GUID as tiebreaker

Do **not** key behavior on hardcoded GUIDs (ROADMAP Section 4: resolve display names at runtime via `getPolicyTypes`, `PolicyApi.d.ts:16`). Kind detection order:

1. **Settings-shape discrimination** (primary): `'minimumApproverCount' in s && !('requiredReviewerIds' in s)` -> minimumReviewers; `'buildDefinitionId' in s` -> build; `'requiredReviewerIds' in s` -> requiredReviewers; `('allowSquash' in s || 'useSquashMerge' in s || 'allowNoFastForward' in s || 'allowRebase' in s || 'allowRebaseMerge' in s)` -> mergeStrategy.
2. **GUID tiebreaker** (only for the two shape-less types, comment resolution vs work-item linking, which both carry only `scope`): compare `configuration.type.id` against the well-known GUIDs, declared as documented defaults:

```ts
/** Well-known cloud GUIDs (ROADMAP Section 4: "confirm per-org"). Used ONLY to disambiguate the two
 *  settings-shapeless policy types; display names always come from getPolicyTypes/type.displayName. */
export const WellKnownPolicyTypeIds = {
	minimumReviewers: 'fa4e907d-c16b-4a4c-9dfa-4906e5d171dd', // ROADMAP Section 4
	workItemLinking: '40e92b44-2fe1-4dd6-b3d8-74a9c21d0c6e', // ROADMAP Section 4
	commentRequirements: 'c6a1889d-b943-4856-b76f-9e46bb6b0df2', // docs-derived; verify per-org
} as const;
```

3. **Fallback**: anything unmatched -> kind `'other'`, rendered with its runtime display name and no per-type detail. This keeps unknown/custom/status-check policy types visible instead of dropped.

Display names: cache `getPolicyTypes(project)` (`PolicyApi.d.ts:16`) once per `AzdoRepository` instance (same caching idiom as `_metadata`, `azdoRepository.ts:88-110`) into a `Map<typeId, displayName>`; fall back to `configuration.type.displayName` (`PolicyInterfaces.d.ts:142`, populated on embedded configs) and finally to `'Branch policy'`.

## 3. Auto-complete state machine (AC-02)

### 3.1 States (derived per render from `{ autoCompleteSetBy, currentUser, policies, mergeable, state, isDraft, hasWritePermission }` - nothing new is stored)

```
                      set (me)
  +--------+  ----------------------->  +------------+
  |  NONE  |                            | SET_BY_ME  |----+
  |        |  set by other (observed    +------------+    |  cancel (me/other)
  |        |  via poll/refresh)                           +--------------------> NONE
  |        |  ----------------------->  +--------------+  |
  +--------+                            | SET_BY_OTHER |--+
      |                                 +--------------+
      |                                       |
      | complete now                          | all blocking policies pass
      | (mergeable=Succeeded)                 | (server completes; observed via
      v                                       v  poll or action reply)
  +-----------+ <-----------------------------+
  | COMPLETED |
  +-----------+
```

- **NONE**: `!autoCompleteSetBy` (or id is the zero GUID). Two sub-renders:
  - `NONE / completable` - no pending/failed blocking policies AND `mergeable === Succeeded`: today's Merge button unchanged (`merge.tsx:196-212`); no auto-complete affordance (matches ADO web, where auto-complete only appears while something is pending).
  - `NONE / blocked` - `pendingBlockingPolicies(policies).length > 0` (Section 4): primary button **Set auto-complete** (replacing today's dead end where `PrActions` renders nothing, `merge.tsx:214-227`), opening `ConfirmAutoComplete` - the `ConfirmMerge` form (`merge.tsx:285-338`) generalized with: strategy select (`MergeSelect`), delete-branch + transition-work-items checkboxes, optional merge-commit-message textarea (pre-filled `Merged PR {number}: {title}`), submit posting `pr.set-autocomplete { enable: true, options }`.
- **SET_BY_ME**: `autoCompleteSetBy.id === currentUser.id` (overview has `currentUser: Identity` in cache, `cache.ts:68`). Banner: "Auto-complete set by you" + options summary line ("Squash, delete branch, complete work items") + **Cancel auto-complete** button.
- **SET_BY_OTHER**: `autoCompleteSetBy` set, id differs from mine. Banner: "Auto-complete set by {autoCompleteSetBy.name}" + same options summary + **Cancel auto-complete** gated on `hasWritePermission` (ADO permits anyone with contribute rights to cancel). The activity-bar simple view lacks `currentUser` in its payload (`activityBarViewProvider.ts:136-168` posts `isAuthor` only); the banner component treats missing `currentUser` as "show name, never 'you'" - cosmetic degradation only.
- **COMPLETED**: `state === Completed` - existing "Pull request successfully merged." branch (`merge.tsx:64-68`).

While `SET_BY_*` and state is Active, the merge form is hidden (server owns completion); an "Update options" affordance is **deferred** - cancel-then-re-set covers it (two clicks, zero extra states).

### 3.2 The becomes-completable race (and friends)

1. **Set-completes-immediately**: if every blocking policy is already satisfied when the PATCH lands, the server completes the PR during the same operation. Handling: the `pr.set-autocomplete` reply carries `state` from the PATCH response; the webview treats the reply as authoritative (`updatePR(reply)`) and lands directly in COMPLETED. Never assume the reply state is Active.
2. **Cancel-vs-complete**: user clicks Cancel just as the server completes. The cancel PATCH fails (PR no longer Active). Host catches, re-fetches via `getPullRequestById` (`GitApi.d.ts:86`), and replies with the fresh state + a non-error toast "Pull request was already completed by auto-complete." The webview transitions to COMPLETED, not an error state.
3. **Concurrent set-by-other**: last-writer-wins on the server; our reply reflects whatever the PATCH returned (`autoCompleteSetBy` in the response body). Poll/refresh converges the banner. No client-side locking.
4. **Push/vote resets a policy while set**: auto-complete stays armed server-side; policy rows revert to Running via the poll; banner unchanged. No action needed - document in the component comment so nobody "fixes" it.
5. **Poll observes completion**: while auto-complete is set, the Section 2.4 poll keeps running; when a `pr.checkPolicies`/`pr.checkMergeability` cycle or `pr.refresh` shows `state: Completed`, render COMPLETED. (The existing mergeability poll self-limits to NotSet/Queued, `merge.tsx:108-116` - the auto-complete-armed condition is the new keep-alive for the policy poll only.)

## 4. AC-02 reuses POL-01's pending-blocking-policies signal (ROADMAP Section 5.2: POL-01 lands first)

Single shared selector, exported from `webviews/components/merge.tsx` (or a small `webviews/common/policyUtil.ts` if merge.tsx gets crowded):

```ts
import { PolicyEvaluationStatus } from 'azure-devops-node-api/interfaces/PolicyInterfaces';

export function pendingBlockingPolicies(policies?: PullRequestPolicyEvaluation[]): PullRequestPolicyEvaluation[] {
	return (policies ?? []).filter(
		p =>
			p.isBlocking &&
			(p.status === PolicyEvaluationStatus.Queued ||
				p.status === PolicyEvaluationStatus.Running ||
				p.status === PolicyEvaluationStatus.Rejected ||
				p.status === PolicyEvaluationStatus.Broken),
	);
}
```

Consumers:

- **POL-01 summary row**: `pendingBlockingPolicies(pr.policies).length` drives "N blocking policies not satisfied" vs "All branch policies passed".
- **AC-02 show/hide**: the `NONE / blocked` sub-state (Section 3.1) is `pendingBlockingPolicies(pr.policies).length > 0` - Set auto-complete appears exactly when the POL-01 summary says something is unsatisfied. One selector, one truth; the two features can never disagree about whether the PR is blocked.
- **Degradation rule**: `policies === undefined` (fetch unavailable/failed) makes the selector return `[]`, so AC-02 falls back to today's behavior (merge button only when `mergeable === Succeeded`) and no auto-complete button renders. Auto-complete is strictly additive on orgs where evaluations can't be read.
- Rejected/Broken are deliberately included as "pending" for both consumers: a rejected build validation still auto-completes after a re-run passes, so Set auto-complete must be offered there too - this matches ADO web.

POL-04's Re-run lives on rows the POL-01 panel already renders; it adds no new signal. AC-04 feeds `mergeMethodsAvailability`, which both `ConfirmMerge` and `ConfirmAutoComplete` consume through the same `MergeSelect` (`merge.tsx:353+`) - set once in the host, both forms inherit it.

## 5. AC-04: merge strategies restricted by "Limit merge types" policy

**Where:** replace the hardcoded `getPullRequestRepositoryAccessAndMergeMethods` (`src/azdo/folderRepositoryManager.ts:1343-1353`, currently returns all-true) with a policy-aware implementation. `hasWritePermission: true` stays hardcoded - permissions are out of v1.5 scope.

**Fetch (primary - REST, because the branch-scoped route is a verified 10.2.2 hole, Section 0):**
`GET {orgUrl}/{projectId}/_apis/git/policy/configurations?repositoryId={repoId}&refName={targetRefName}&api-version=7.1` - server-side branch matching including Prefix and DefaultBranch scopes. Implement as a small typed wrapper `src/azdo/gitPolicyConfigApi.ts` following the existing raw-REST precedent `src/azdo/entitlementApi.ts:50-100` (extends `ClientApiBase`, uses `webApi.authHandler` + `_getResourceAreaUrl`). `targetRefName` is the PR's full target ref (`refs/heads/...`), URL-encoded by the REST client.

**Fallback (client-side, if the REST route errors):** `PolicyApi.getPolicyConfigurations(projectId)` (`PolicyApi.d.ts:8`) + client-side scope filtering against `settings.scope[]` (Section 1.1): match `repositoryId` (null matches all), then `matchKind` Exact (case-insensitive full ref compare), Prefix (ref startsWith), DefaultBranch (compare against repo default branch from metadata). Log which path served the result.

**Mapping to `MergeMethodsAvailability` (`src/azdo/interface.ts:189-191`):**

- No merge-strategy policy matched -> all four `true` (current behavior preserved).
- One or more matched (`isEnabled && isBlocking`): availability = logical **AND** across policies of each `allow*` flag (a strategy is offered only if every applicable policy allows it) - conservative, mirrors server enforcement. Legacy `useSquashMerge: true` maps to `{ Squash: true }` only.
- All-false result (pathological but possible with overlapping policies): keep all four **enabled** and log a warning - better to let the server reject with a real error than to render a form with zero options. [judgment call - revisit if it ever fires]

**Default selection:** `getDefaultMergeMethod(availability, preferred)` (`src/azdo/pullRequestOverview.ts:1018`) already picks the user's preferred method only when available and falls back across the availability map - no change needed once it receives real availability. The `(not enabled)` disabled-option rendering in `MergeSelect` (`merge.tsx:353+`) also just starts working.

**Call sites:** both hosts already call `getPullRequestRepositoryAccessAndMergeMethods` in their init `Promise.all` (`pullRequestOverview.ts:192`, `activityBarViewProvider.ts:98-104`) - no plumbing changes, only the implementation. Cache the result per (repoId, targetRef) on FolderRepositoryManager with a short TTL (policies change rarely; refetch on `pr.refresh`).

**Auto-complete interplay:** `ConfirmAutoComplete` uses the same `MergeSelect`, so a strategy forbidden by policy can't be armed into `completionOptions` - that's the "also feeds the auto-complete form" requirement of AC-04.
