## 1. UX-05, Design tokens: color, spacing, radius, theming

### 1.1 What's wrong today (verified)

- Buttons and submits are hard-squared: `border-radius: 0px` (`webviews/common/common.css:17`), as are
  `<select>`s (`common.css:68`). Meanwhile other elements already use 2px (`.work-item`,
  `editorWebview/index.css:1036`), 3px (`pre`, `index.css:727`; mobile reviewer chips,
  `index.css:935-940`), 4px (`#status` badge, `index.css:405`; `.required-badge`, `common.css:161`),
  and 10px (`.prIcon`, `index.css:346`). Five radii, no system.
- Hardcoded colors that ignore the theme: `hr { border-top: 1px solid #555 }` (`index.css:95`) and the
  `.vscode-light`/`.vscode-dark` rgba() overrides for code blocks and table borders
  (`index.css:742-790`). These approximate what `--vscode-*` variables already provide exactly.
- Spacing is ad-hoc per rule: margins/paddings of 4/5/6/8/10/12/15/16/19/20px all appear
  (e.g. `index.css:176-178` `margin-top: 5px; margin-left: 15px`, `index.css:630` `padding: 19px 0`).
- The only "semantic color" in the panel is whatever the bundled SVGs carry (VOTE-07 icons) plus
  `--vscode-badge-*` on two badges. Status text (policy failures, merge conflicts, "successfully
  merged") renders in default foreground: a failed policy row and a passed one differ only by icon.
- Structural borders lean on `--vscode-list-inactiveSelectionBackground` (comment headers
  `index.css:142-143`, comment bodies `index.css:291`, `#status-checks` `index.css:230`, title rule
  `index.css:59`). That token is a _selection_ color, not a border color; in several themes it's
  near-invisible or oddly tinted. It reads as "flat gray" precisely because it isn't a border token.

### 1.2 Token sheet

Add a token block at the top of `webviews/common/common.css` (shared by both webviews, since the
editor webview and activity-bar webview both import it via their bundles). All tokens alias VS Code
theme variables, so light/dark/high-contrast come for free; we never hardcode hex.

```css
:root {
	/* -- radius -- */
	--azdo-radius-sm: 2px; /* chips, inline badges, code spans */
	--azdo-radius-md: 4px; /* buttons, inputs, selects, badges */
	--azdo-radius-lg: 6px; /* cards: comment containers, status panel, sidebar cards */

	/* -- spacing (4px base) -- */
	--azdo-space-1: 4px;
	--azdo-space-2: 8px;
	--azdo-space-3: 12px;
	--azdo-space-4: 16px;
	--azdo-space-5: 20px;
	--azdo-space-6: 24px;

	/* -- surfaces & borders -- */
	--azdo-surface: var(--vscode-editorWidget-background, var(--vscode-editor-background));
	--azdo-surface-header: var(--vscode-sideBarSectionHeader-background, transparent);
	--azdo-border: var(--vscode-widget-border, var(--vscode-panel-border, rgba(128, 128, 128, 0.35)));
	--azdo-border-subtle: var(--vscode-editorWidget-border, var(--azdo-border));

	/* -- semantic status -- */
	--azdo-success: var(--vscode-charts-green, var(--vscode-testing-iconPassed));
	--azdo-danger: var(--vscode-errorForeground);
	--azdo-warning: var(--vscode-charts-yellow, var(--vscode-editorWarning-foreground));
	--azdo-info: var(--vscode-charts-blue, var(--vscode-textLink-foreground));
	--azdo-muted: var(--vscode-descriptionForeground);
}
```

Rationale for the choices:

- **`--vscode-charts-*`** is the theme-adaptive palette VS Code itself uses for colored indicators; it
  is defined in both default light and dark themes and tracks custom themes. Fallbacks chain to
  always-defined tokens so a theme that omits `charts.*` still renders sanely.
- **`--vscode-widget-border` / `--vscode-panel-border`** replace every
  `--vscode-list-inactiveSelectionBackground` border. The high-contrast overrides at
  `index.css:761-770` already do exactly this; the token sheet just makes it the rule everywhere
  instead of the HC exception.
- **`--vscode-editorWidget-background`** gives cards a one-step-off-editor surface in both light and
  dark themes without inventing our own gray.
- Radius stops at 6px: VS Code's own UI (quick pick, notifications, hovers) sits in the 3-6px range;
  anything rounder looks foreign inside the editor.

### 1.3 Application rules (mechanical)

| Rule                                                                             | Replace                                                                                              | With                                                                                                                                                                                                                         |
| -------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Buttons/inputs/selects                                                           | `border-radius: 0px` (`common.css:17,68`)                                                            | `var(--azdo-radius-md)`                                                                                                                                                                                                      |
| All structural borders                                                           | `1px solid var(--vscode-list-inactiveSelectionBackground)` (`index.css:59,143,230,291,548,794,1007`) | `1px solid var(--azdo-border-subtle)`                                                                                                                                                                                        |
| `hr`                                                                             | `#555` (`index.css:95`)                                                                              | `var(--azdo-border-subtle)`                                                                                                                                                                                                  |
| Code-block/table theming                                                         | `.vscode-light`/`.vscode-dark` rgba blocks (`index.css:742-790`)                                     | `var(--vscode-textCodeBlock-background)` for pre/code, `var(--azdo-border-subtle)` for table borders; delete the per-theme-class rules                                                                                       |
| Card containers (`#status-checks`, comment containers, sidebar cards, section 3) | square, mixed padding                                                                                | `border-radius: var(--azdo-radius-lg); padding: var(--azdo-space-3)`                                                                                                                                                         |

Keep the existing `.vscode-high-contrast` blocks (`common.css:172-185`, `index.css:872-900`); they
already do the right thing and the token fallback chain doesn't cover HC's no-fill-buttons rule.

Sizing note: `#app { grid-template-columns: 670px auto }` (`index.css:7-9`) fixes the _main_ column
and lets the sidebar absorb leftovers, which is backwards. Change to
`grid-template-columns: minmax(0, 1fr) minmax(220px, 280px)` with `column-gap: var(--azdo-space-5)`
(drop `#sidebar { padding-left: 20px }`, `index.css:22-26`). The existing `@media (max-width: 925px)`
single-column fallback (`index.css:902`) stays.

## 2. UX-02, Completed/abandoned PRs: read-only summary presentation

### 2.1 What renders today on a finished PR (verified live + source)

- **Checkout**: `CheckoutButtons` renders for any non-issue, any state (`header.tsx:184-189`); the
  only gate is `isCurrentlyCheckedOut`.
- **VotePanel + Cast Vote**: `Sidebar` always renders it (`sidebar.tsx:21`); no state gate.
- **Add-reviewer "+"**: `ReviewerPanel` gates only on `hasWritePermission` (`sidebar.tsx:99-109`), as
  does the Work Items "+" (`sidebar.tsx:39-48`) and the hover-delete on reviewers/work-items
  (`reviewer.tsx:31-38`, `sidebar.tsx:70-78`).
- **"Close Pull Request"** in the comment form is the one control that _is_ already state-gated
  (`comment.tsx:412`, `disabled` when `state !== Active`), but it renders disabled instead of
  disappearing.
- What already handles the finished state: `StatusChecks` swaps the whole checks/merge block for
  "Pull request successfully merged." plus `DeleteBranch` (`merge.tsx:72-84`; note the "abondoned"
  typo at `merge.tsx:82`, fix it in this pass), and the header badge/verb (`header.tsx:195-211`).

### 2.2 Design

Derive once, pass down: `const isActive = pr.state === PullRequestStatus.Active` (compute in
`Overview`, `webviews/editorWebview/overview.tsx:16`, and thread as a prop, or read `pr.state` in each
component). Prefer the prop so the activity-bar view (`webviews/activityBarView/`) reuses the same
logic when it renders shared components.

**When `!isActive`:**

| Surface                          | Active PR                             | Finished PR                                                                                                                                                                                                              |
| -------------------------------- | ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Header buttons                   | Checkout / Exit Review Mode + Refresh | Refresh only. If `isCurrentlyCheckedOut`, keep **Exit Review Mode** (users must be able to leave review mode of a merged PR); never offer plain Checkout.                                                                |
| Title action bar                 | Edit / Copy Link / Convert to draft   | Copy Link only (`canConvertToDraft` already checks Active, `header.tsx:71`; add the same gate to title Edit)                                                                                                             |
| Sidebar vote panel               | VotePanel                             | **Removed.** Replaced by the outcome line inside the summary card (section 2.3).                                                                                                                                         |
| Reviewer sections                | +add, hover-remove, vote per row      | Same rows, final votes kept (they are the review record), no "+" button, no hover-remove                                                                                                                                 |
| Work items                       | +add, hover-remove                    | List only, no add/remove                                                                                                                                                                                                 |
| Comment form                     | Comment + Close Pull Request          | Keep the comment box (ADO allows commenting on completed PRs and it round-trips fine; verified this session that comment create works via `pr.comment`); drop the Close button entirely instead of rendering it disabled |
| Checks/merge block               | Policy + status + merge actions       | Already handled (`merge.tsx:72-84`): keep outcome banner + Delete branch (Completed) / banner only (Abandoned)                                                                                                           |

**Checkout replacement:** nothing. AC-08 already gives Delete branch on completed PRs, and the merged
commit is reachable from the timeline's merge event. A "View merge commit" link is only possible when

## 3. UX-01, Reviewer/vote sidebar

### 3.1 What's wrong today (verified)

- Your current vote is invisible: `VoteSelect` renders "(current vote)" _only as option text inside
  the closed dropdown_ (`sidebar.tsx:168-177`), and it's `defaultValue`-driven, so after casting the
  select doesn't move and nothing on screen says what your standing vote is.
- Cast Vote's only feedback is `disabled={vote.toString() === selectedVote}` flipping
  (`sidebar.tsx:147`). Live test: clicking Cast Vote produced no visible change (the vote _did_
  register server-side and `appendReview` did update the reviewer rows, since `appendReview` replaces
  `state.reviewers` in `context.tsx`, but your own row is easy to miss and there's no confirmation
  moment).
- No busy state: `castVote` awaits `votePullRequest` with no spinner/label change
  (`sidebar.tsx:134-136`).
- Layout: `.vote` gets `margin-bottom: 20px` and the select 5px (`index.css:28-40`); the button is
  `margin-left: auto` on its own row. No grouping, no heading; the vote UI floats above "Required
  Reviewers" with nothing binding it.
- The `'0': 'Reset Vote'` action sits in the same dropdown as real votes (`sidebar.tsx:117-123`),
  which is how "Reset Vote" ends up looking like a fifth vote value (reviewer.tsx already had to work
  around this for state display, `reviewer.tsx:58-65`).

### 3.2 Design

Replace `VotePanel` with a **"Your review" card**, same card styling as section 2.3, containing three
rows:

```
+----------------------------------+
| YOUR REVIEW                      |  <- section label, --azdo-muted, 11px uppercase
| [check] Approved with suggestion |  <- current-vote row: VOTE-07 icon + VOTE_STATE_TEXT, 13px bold
| [ Approve            v][ Vote ]  |  <- controlled select + primary button, one flex row, space-2 gap
| Reset vote                       |  <- link-styled button; only when vote != 0
+----------------------------------+
```

Decisions, in order of user impact:

1. **Always-visible current-vote row.** Icon + text via the existing `REVIEW_STATE_ICON` /
   `VOTE_STATE_TEXT` maps; export them from `reviewer.tsx` instead of duplicating (today
   `reviewer.tsx:50-65` owns them privately). When vote is 0 it reads "(circle) Not voted yet" in
   `--azdo-muted`. This is the unmistakable indicator the roadmap asks for: it does not require
   opening anything, and it's the same glyph the reviewer row shows, so the two can't disagree.
2. **Controlled select, actions only.** `value={selectedVote}` + `onChange` (replacing `defaultValue`,
   `sidebar.tsx:169`) listing only the four real votes ('10','5','-5','-10'); drop the "(current
   vote)" option-suffix hack entirely, the row above owns that job. "Reset Vote" moves out of the
   select to a small link-button under the row (calls `castVote('0')`), shown only when you have a
   standing vote.
3. **Cast feedback, three phases.**
   - _Busy:_ button text "Casting...", `disabled`, `aria-busy="true"` while the promise is in flight
     (state in `VotePanel`, try/finally like every other busy pattern in this codebase, e.g.
     `merge.tsx:329-341`).
   - _Success:_ the current-vote row updates (it re-derives from `pr.reviewers` via `sidebar.tsx:21`,
     which `appendReview` refreshes, so no new plumbing) **and** a transient inline confirmation
     replaces the button label for ~2.5s: "(check) Vote recorded" in `--azdo-success`. Inline text,
     not a toast: the webview has no toast infrastructure and the confirmation belongs next to the
     control that caused it. Wrap the row in `aria-live="polite"` so screen readers announce it.
   - _Failure:_ `postMessage` rejections currently vanish (the `pr.vote` handler replies-with-error,
     but `votePullRequest` doesn't catch). Catch in `castVote`, render "Vote failed, try again" in
     `--azdo-danger` in the same slot.
4. **Button enablement** stays "selected differs from current" but now derived from the _live_ vote
   (`pr.reviewers`-sourced prop) vs the controlled select value; the current disabled logic
   (`sidebar.tsx:147`) already means this, and keeping it plus the visible current-vote row makes the
   _why_ of a disabled button legible for the first time.
5. **Section rhythm below:** `Required Reviewers` / `Optional Reviewers` / `Work Items` headers get
   the same 11px uppercase muted treatment; empty categories render "None" in `--azdo-muted` instead
   of collapsing to zero height (today an empty required list is just absent, `sidebar.tsx:111-113`).
   The `.section-item .login { width: 129px }` truncation column (`common.css:111-114`) becomes
   `flex: 1; min-width: 0` with ellipsis so long display names and the vote status share space
   predictably.

**Author case:** ADO lets PR authors vote on their own PRs (unless a policy forbids it), so keep the
card for authors; no special casing.

**Key files:** `webviews/components/sidebar.tsx`, `webviews/components/reviewer.tsx` (export maps),
`webviews/common/common.css`, `webviews/editorWebview/index.css`.

## 4. UX-03, Comment section design pass

This is the judgment-heavy one. Scope: the overview webview's description, timeline threads, and
composers (`comment.tsx`, `timeline.tsx`, `editorWebview/index.css`). The _editor-margin_ comment
experience (`src/view/reviewCommentController.ts:32`) is deliberately not restyled here; it renders
through VS Code's native Comment API and its resolve/reactivate affordances are THR-01/THR-02 (v1.7)
work. This section only has to not fight it: same status vocabulary, same colors.

### 4.1 Current anatomy (verified)

- **Comment card** = `CommentBox`: header strip with avatar/author/timestamp on a
  `--vscode-list-inactiveSelectionBackground` fill + 1px same-color border (`index.css:133-144`),
  body in a separate bordered box with `border-top: none` (`index.css:288-293`). Two stacked boxes,
  selection-colored borders: this is the "dated bubble styling."
- **Thread** = `CommentEventView` (`timeline.tsx:200-255`): optional file-path diff header, then
  every comment full-width (replies indented 20px via
  `.thread-container .comment-container ~ .comment-container`, `index.css:996-999`), then a
  right-aligned bare "Reply" button (`index.css:1001-1003`).
- **Thread status** = a raw `<select>` crammed into the _first comment's header_ between timestamp and
  actions (`comment.tsx:154-162`), rendered only when `isFirstCommentInThread`, which is computed as
  `c.id === 1` (`timeline.tsx:238`), i.e. relies on ADO comment IDs starting at 1 per thread.
- **Status meaning is invisible:** Active vs Fixed vs WontFix vs Closed all render identically except
  for the select's text. No color, no collapsed state, no filtering. The only rollup is the header's
  "8/9 comments resolved" line (`header.tsx:213-222`).
- **Timeline order is newest-first** (`timeline.tsx:33` sorts descending by `publishedDate`), so the
  conversation reads bottom-up while each thread reads top-down, and the composer (`AddComment`) sits
  _above_ the timeline (`overview.tsx:43-44`): you type at the top and your new thread appears
  directly underneath, but the overall page reads in reverse.
- **Hover-only actions:** edit/quote appear on `onMouseEnter` (`comment.tsx:79-95`), invisible to
  keyboard users and undiscoverable.
- **Code blocks render in dracula, hardcoded** (`comment.tsx:9-10,257-263`): purple-on-dark inside a
  light-themed VS Code.
- **Composers:** plain `<textarea>` + Cancel/Save (`comment.tsx:233-243,300-342`), no formatting
  affordances, no Ctrl+Enter hint (the handler exists, `comment.tsx:215-223`, it's just secret).

### 4.2 Design

#### 4.2.1 Comment card

One card, not two boxes:

```
+--------------------------------------------+  --azdo-surface, --azdo-border-subtle,
| (av) Corey Yost . 2 days ago    [.. edit ] |  --azdo-radius-lg; header transparent,
| ------------------------------------------ |  no fill, just a subtle bottom rule
| Body: markdown, themed code blocks         |
+--------------------------------------------+
```

- Kill the header fill (`index.css:142`); the card outline now does the grouping. Header keeps avatar
  (24px), bold author, muted relative timestamp; actions live on the right.
- **Actions visible on hover _and_ focus-within**, not mouse-only: keep the `showActionBar` state but
  add a CSS `:focus-within` fallback (`.comment-container:focus-within .action-bar { display: flex }`)
  so keyboard users can reach edit/quote. Buttons get `--azdo-radius-sm` hover backgrounds
  (`--vscode-toolbar-hoverBackground`) instead of today's transparent-on-transparent.
- **Code blocks:** replace the dracula import with theme-conditional styles: `prism`/`oneLight` for
  `.vscode-light` bodies and `vscDarkPlus`/`oneDark` otherwise (both ship in
  `react-syntax-highlighter/dist/esm/styles/prism`; pick at render time from `document.body.className`,
  which VS Code stamps with `vscode-light`/`vscode-dark`/`vscode-high-contrast`). Also set the
  container background to `var(--vscode-textCodeBlock-background)` so inline and block code agree.

#### 4.2.2 Thread container

Threads become one visual unit with a left rail:

```
+ file/path.ts:42 ------------- [Active v] +  <- thread header: path chip (opens diff) + status
| + root comment card ------------------+ |
| | ...                                  | |
| + reply card (rail-indented, no border | |
| |   between siblings, one rule)        | |
| +--------------------------------------+ |
| [Reply...]                              |  <- ghost text-field-looking button, left-aligned
+------------------------------------------+
```

- The **thread owns the border/card**; individual comments inside lose their outer borders and are
  separated by single `--azdo-border-subtle` rules. Replies keep the existing 20px indent but gain a
  2px left rail in `--azdo-border-subtle` so depth is visible even across a page fold.
- **Thread status moves from the first comment's header to the thread header**, next to the file-path
  chip. Mechanically: render the status control in `CommentEventView` (which already owns
  `thread.status` and `onThreadStatusChange`, `timeline.tsx:216-218,239-240`) instead of threading
  `isFirstCommentInThread` into `CommentView`, and delete the fragile `c.id === 1` computation
  (`timeline.tsx:238`) along with the `statusProps` plumbing (`comment.tsx:45-47`).
- **Status control = colored pill + native select behind it.** A `<select>` styled as a pill (status
  color dot + label + chevron) keeps the accessibility and z-index simplicity of a native control
  while reading as a modern chip. Colors: Active `--azdo-info`, Pending `--azdo-warning`,
  Fixed/Closed `--azdo-success`, WontFix `--azdo-muted`. Same palette the editor-side THR-01 work will
  inherit.
- **Resolved threads collapse.** Threads whose status is Fixed/WontFix/Closed render collapsed by
  default: thread header + one muted summary line ("3 comments . resolved by Corey") with a chevron
  to expand. This is the single highest-leverage change for long PRs: resolved noise disappears but
  stays one click away. State is per-render (component `useState`, default collapsed when resolved);
  no persistence needed in this pass.
- **Reply** stays inside the thread footer, but as a full-width ghost input look-alike ("Reply...")
  that swaps to the real composer on click, replacing the right-aligned lone button
  (`timeline.tsx:243-252`, `index.css:1001-1003`). `ReplyToThread` itself is reused unchanged.

#### 4.2.3 Timeline order and composer position

Keep **newest-first** thread ordering (`timeline.tsx:33`); it matches ADO web's overview and puts
recent activity where the scroll position starts. But make the composer placement coherent with it:
`AddComment` stays above the timeline (`overview.tsx:43-44`), which is _correct_ for newest-first
(your new comment appears directly under the box). No change, but record it as deliberate: do not
"fix" the ordering to oldest-first without also moving the composer to the bottom.

System/commit events in the timeline (`timeline.tsx:66-91`) get one polish pass: muted single-line
rows, 16px icon, timestamp right-aligned. They already are close (`index.css:449-516`); mostly they
inherit the spacing scale.

#### 4.2.4 Composers

- Add a **one-line footer hint** to all three composers (AddComment, ReplyToThread, EditComment):
  "Markdown supported . Cmd/Ctrl+Enter to submit" in `--azdo-muted` 11px. The keybinding already works
  (`comment.tsx:215-223,321-329,378-385`); it just needs to be discoverable.
- **No formatting toolbar in this pass.** A real toolbar (bold/italic/code/suggestion) is meaningful
  scope with selection-manipulation code, and DLA-07 (image paste) will force a composer rework
  anyway, so do the toolbar there, once, rather than twice. This is the one place section 4
  deliberately under-delivers vs "modern UX overhaul"; flagged as the follow-on item in section 7.
- Primary submit buttons become visually primary (they already use `--vscode-button-background`; the
  Cancel/secondary spacing tightens to `--azdo-space-2`).
- The `AddComment` "Close Pull Request" button: gated per section 2.2 (rendered only when Active); it
  stays the leftmost, `secondary`-styled action (`comment.tsx:408-418`).

#### 4.2.5 Description block

The PR description reuses `CommentView` with `isPRDescription` (`overview.tsx:26-42`) and inherits the
card restyle for free. One addition: when `pr.body` is empty the "No description provided" placeholder
(`comment.tsx:269-275`) sits in a dashed-border card in `--azdo-muted`, an affordance that this is
fillable (Edit lives in the hover actions, canEdit permitting).

## 5. UX-04, Panel reuse vs one tab per PR: recommendation

**Recommendation: switch to one panel per PR, keyed by PR ID, with reveal-if-open.** Not a setting,
not a discussion item: change the default.

Current: `PullRequestOverviewPanel.currentPanel` is a static singleton, so any open request reveals
and overwrites the existing panel (`src/azdo/pullRequestOverview.ts:40,72-87`), retitling it via
`update()`. Upstream GitHub-extension behavior, inherited.

Why per-PR panels win:

1. **It's how VS Code treats every other document.** Clicking two files gives two tabs; users manage
   tab volume with the editor's own machinery (preview tabs, close-others, pinning). A webview that
   silently repurposes itself is the surprising behavior, and it's exactly what the user feedback
   flagged.
2. **Reuse destroys context mid-review.** Scroll position, expanded threads, an in-progress vote
   selection, half-written top-level comment: all discarded when a second PR is clicked (only comment
   drafts survive, via `pendingCommentDrafts`). Comparing two PRs, the most common reason to open a
   second one, is impossible today without the browser.
3. **The cost asymmetry favors tabs.** Tab clutter is user-correctable in one click; lost review state
   is not correctable at all. And realistic concurrent-PR counts (2-4) are trivial tab loads.

Mechanics (S-sized):

- `static currentPanel?` becomes `static panels = new Map<number, PullRequestOverviewPanel>()` keyed
  by `pr.getPullRequestId()`. `createOrShow` reveals the mapped panel if present (then still calls
  `update()` to refresh), else constructs.
- `dispose()` removes itself from the map (today it nulls `currentPanel` via the `_currentPanel`
  setter, `pullRequestOverview.ts:89-91`; same hook, map-delete instead).
- `refresh()` (`pullRequestOverview.ts:93-97`) iterates `panels.values()`; per-PR message routing
  already works because each panel owns its own `_panel`/listeners. The singleton was never
  load-bearing for correctness, only for reuse.
- Audit `currentPanel` call sites (`getCurrentTitle`, activation code); grep shows only
  registration/refresh paths, each becomes a map lookup or iteration.
- The active-PR _activity-bar_ view is unaffected (separate `WebviewViewProvider`).

Escape hatch: if tab pressure materializes in practice, the later refinement is opening PRs in
_preview mode_ (italic tab title, reused until pinned) via `vscode.window.createWebviewPanel`'s
`preserveFocus` plus the workbench's preview semantics, not a return to the singleton. Ship the map
first; only add a `azdoPullRequests.openPullRequestsInNewTab: false` compat setting if someone
actually asks.

**Key file:** `src/azdo/pullRequestOverview.ts` (plus call-site audit).
